## 1. 가장 추천하는 커밋 단위

문제 하나 = 커밋 하나

예:
```
feat: BOJ 2563 색종이 풀이 추가
```
이 방식의 장점

- 문제별 히스토리 확인 가능

- 복습할 때 찾기 쉬움

- Git log가 깔끔함


## 2. 커밋 메시지 포맷
```
[플랫폼] 문제번호 문제이름 - 알고리즘
```
예시
```
[BOJ] 2563 색종이 - brute force
```
또는
```
[BOJ] 2178 미로탐색 - BFS
```

## 3. 시행착오도 커밋 남기기
- 출력초과

- 시간초과

- 다른 접근 시도

예:
```
[BOJ] 2563 색종이 - sweep line 시도
[BOJ] 2563 색종이 - brute force 풀이로 수정
```

## 4. 코드 수정 커밋

예
```
refactor: BOJ 2563 코드 정리
```
또는
```
docs: BOJ 2563 풀이 설명 추가
```

## 5. 커밋 메시지 타입
| 타입       | 의미        |
| -------- | --------- |
| feat     | 문제 풀이 추가  |
| fix      | 버그 수정     |
| refactor | 코드 정리     |
| docs     | README 작성 |

### 예시 Git 로그
```
feat: BOJ 2563 색종이 - sweep line 풀이
fix: BOJ 2563 출력 초과 수정
refactor: BOJ 2563 코드 단순화
docs: BOJ 2563 풀이 설명 추가
feat: BOJ 2178 미로탐색 - BFS 풀이
```

## 6. 커밋 타이밍

```
1 문제 풀이 완료
2 README 작성
3 코드 리팩토링
```
